1.  Introduction
1.1  Purpose of this Document
This Software Requirements Specification defines the complete functional and non-functional requirements for GodownPro — a material and project management system built for Yorks Airconditioning and Refrigeration LLC SPC. It serves as the authoritative reference for design, development, testing, and client acceptance of the MVP.
1.2  The Problem Being Solved
Yorks currently manages all material requests through handwritten slips and verbal communication between site engineers and the store employee. This process creates five critical business problems:
Problem
Business Impact
No visibility of stock for engineers on site
Engineers request items without knowing availability, causing delays and incorrect orders
No approval chain — material leaves store without formal sign-off
No accountability for who authorised what, leading to untracked material loss
No audit trail — handwritten slips get lost or are illegible
Impossible to reconstruct what was issued to which project for costing or disputes
No real-time status — engineer has no idea if request is being processed
Engineers call procurement repeatedly, wasting time on both sides
No project cost data — accountant cannot see material usage per project
Projects regularly exceed material budgets with no data to explain why
1.3  Solution Summary
GodownPro replaces the paper process with a Flutter mobile app for engineers and a Flutter web app for the office, both connected to a single Firebase backend in real time. The system enforces a structured two-phase workflow — Planning before work starts and Execution during site work — with a complete non-deletable audit trail of every action.

5.  Non-Functional Requirements
Non-functional requirements define how the system performs and behaves — not just what it does. Each requirement below includes a measurable target so it can be objectively tested and accepted.
5.1  Performance
ID
Attribute
Requirement
Measurable Target
NFR-001
Response Time
All user actions (submit, approve, dispatch) shall complete and reflect on screen within 2 seconds under normal network conditions.
Tested at 50 concurrent users. 95th percentile response time ≤ 2s.
NFR-002

A status change triggered by one user shall appear on another user's screen within 3 seconds without a manual refresh.
Measured end-to-end via Firestore real-time stream. Target: ≤ 3s.
NFR-003

The mobile app shall be fully interactive within 4 seconds on a standard 4G connection.
Measured from app open to home screen ready. Target: ≤ 4s on 4G.
NFR-004

The web app shall be fully interactive within 3 seconds on a standard office broadband connection.
Measured via Chrome DevTools Lighthouse. Target: Performance score ≥ 80.
NFR-005
Inventory Search
A search query on the inventory shall return results within 1 second for a dataset of up to 1,000 items.
Tested with 1,000 inventory records. Query response ≤ 1s.

ID
Attribute
Requirement
Measurable Target

8.  Glossary
Term
Definition
Phase 1
The planning phase of a project — occurs before site work begins. Engineer creates a material plan, procurement arranges items, engineer gives final approval.
Phase 2
The execution phase — occurs during active site work. Engineers submit fast material requests which procurement dispatches directly.
Material Plan
A list of items the engineer expects to need for a project, submitted in Phase 1.
Material Request
A day-to-day request for items submitted by an engineer during Phase 2 execution.
Auto-tick
Automatic green confirmation applied by the system to items in a Phase 1 plan that exist in inventory with sufficient stock.
Dispatch
The act of procurement confirming and releasing physical materials to be delivered to a site engineer.
Diff View
A screen showing the differences between two versions of a material plan — highlighting what was added, removed, or changed.
RAL Colour
A standardised European colour reference system used in HVAC for grilles, dampers, and diffusers (e.g. RAL 9006 = silver grey).
Audit Trail
A permanent, non-deletable chronological log of every action taken by every user in the system.
Custom Item
A material item created by an engineer that does not yet exist in the inventory, submitted with description fields for procurement to source.
Firestore
Google Firebase's NoSQL real-time document database used as the primary data store.
Firebase Auth
Google Firebase's authentication service used for email/password login.
Flutter
Google's cross-platform UI framework used to build both the mobile and web applications from a single codebase.
